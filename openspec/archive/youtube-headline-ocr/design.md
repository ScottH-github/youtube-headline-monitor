## Context

東森財經新聞（EBC）的「股動錢潮」節目在 YouTube 直播。畫面中會不定時出現黃色橫幅的重要訊息（獨家消息、最新消息），位於畫面右半邊的下方區域。黃色橫幅停留時間不固定，通常超過 3 秒。

黃色橫幅結構：
- 左側：可能有彩色標籤（內容不固定）
- 右側：黃底上深色中文文字
- 偵測依據為黃色大面積區塊，不依賴左側標籤內容

## Goals / Non-Goals

### Goals
- 穩定擷取 YouTube 直播串流，處理斷線重連
- 準確偵測黃色頭條區塊的出現
- 圖像比對去重，同一條新聞只記錄一次
- 中文 OCR 辨識準確率高
- 保留全幀截圖 + 黃色區塊截圖
- 產生可瀏覽搜尋的 HTML 報告

### Non-Goals
- 不做即時推播通知（未來可擴充）
- 不處理黃色橫幅以外的畫面內容
- 第一版不做 Web UI 即時監控介面

## Decisions

### 1. 串流擷取：yt-dlp + OpenCV
- **選擇**：yt-dlp 取得串流 URL → OpenCV VideoCapture 讀取
- **替代**：selenium 開瀏覽器截圖
- **理由**：直接讀串流效能最好，不需要 GUI

### 2. 取樣頻率：每 1-2 秒一幀
- **理由**：黃色橫幅停留 >3 秒，1 秒間隔不會漏。HSV 偵測很輕量，但沒必要每秒做 30 次

### 3. 偵測範圍：右半畫面
- **選擇**：裁切右半（x: 50%~100%）再做 HSV 偵測
- **理由**：黃色橫幅固定出現在右半邊，縮小掃描範圍提升效能

### 4. 色彩偵測：HSV 色彩空間
- **選擇**：轉 HSV → 過濾黃色範圍 → 找輪廓 → 面積過濾
- **替代**：RGB 閾值
- **理由**：HSV 對亮度變化更魯棒

### 5. 去重：圖像比對 + 文字比對（兩層）
- **第一層：圖像 SSIM 比對**：每次裁切出黃色區塊後，跟上一張做 SSIM 比對，相似就跳過。極快（<5ms），擋掉連續重複幀
- **第二層：文字歷史比對**：OCR 完成後，跟 SQLite 中所有歷史記錄做文字相似度比對（>80% 視為重複），防止同一條新聞隔段時間重播時被重複記錄

### 6. OCR：落地 LLM（192.168.0.242:8081）
- **選擇**：將黃色區塊截圖送至落地 LLM 進行多模態辨識
- **替代**：EasyOCR（可作為 fallback）
- **理由**：LLM 不只能辨識文字，還能理解上下文、處理排版不規則的情況，準確度更高
- **API**：`POST http://192.168.0.242:8081/v1/chat/completions`（OpenAI compatible，llama.cpp server）
- **模型**：`gemma-4-26B-A4B-it-UD-Q4_K_M.gguf`（Gemma 4 26B，支援多模態圖片辨識）
- **注意**：Gemma 4 有 thinking 模式，回應中 `content` 為實際回答，`reasoning_content` 為思考過程（忽略）。需給足 max_tokens（建議 500+）以確保 thinking 完後有足夠 token 產出回答

### 7. 存檔：SQLite + 截圖檔案
- **選擇**：SQLite 記錄 + 全幀 PNG + 黃色區塊 PNG
- **理由**：輕量、單檔、方便查詢與產生 HTML

## Risks / Trade-offs

- **串流中斷**：YouTube 直播可能斷線 → 自動重連機制
- **HSV 閾值**：影像壓縮可能影響黃色判定 → 參數可調
- **OCR 準確度**：壓縮影像可能降低辨識率 → 用 720p 以上串流
- **EasyOCR 首次載入慢**：約 10 秒 → 只載入一次，之後複用
